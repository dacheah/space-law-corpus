# 06 — Two-Layer Separation (concrete schema)

**Status:** APPROVED — locked 2026-07-02.
**Date:** 2026-07-02

This document turns the "hard wall" between the authoritative and derived layers into concrete file structure and schema, so the separation is enforced by how the corpus is physically organised — not merely by good intentions.

---

## 1. The wall, in one sentence

**Where a file lives tells you what it is.** Anything under `authoritative/` is primary source text with full provenance. Anything under `derived/` is machine- or human-generated interpretation. No file is ever both, and no derived file is ever presented as authoritative.

## 2. File layout

```
authoritative/
└── <corpus_id>/                         e.g. un/treaty/ost-1967/
    └── <version_id>/                     e.g. 1967-01-27/
        ├── original.<ext>                byte-for-byte as retrieved (pdf/html/…)
        ├── text.txt                      authentic-language faithful rendering (UTF-8, LF)
        └── metadata.yaml                 full provenance record (schema: doc 01)

derived/
└── <corpus_id>/                         SAME id — this is the link back
    └── <version_id>/                     SAME version — derived-of-a-specific-version
        ├── structure.json                articles, sections, dates, parties, citations
        ├── concepts.json                 neutral concept tags
        ├── translations/
        │   ├── en.md                     unofficial machine-draft translation
        │   └── <lang>.md
        ├── summary.md                    plain-language summary (derived, unofficial)
        └── derived-metadata.yaml         provenance of the DERIVATION (schema below)
```

The derived tree mirrors the authoritative tree's `<corpus_id>/<version_id>` path. That mirroring **is** the traceability link: every derived artifact sits at the address of the exact authoritative version it was generated from.

## 3. Derived-layer metadata schema (`derived-metadata.yaml`)

Every derived artifact carries provenance of its *generation*, so model output is fully quarantined and traceable:

```yaml
derived_id: un/treaty/ost-1967/1967-01-27/concepts
derived_type: concept_tags            # translation | summary | structure_extraction | concept_tags
source_corpus_id: un/treaty/ost-1967
source_version_id: "1967-01-27"
source_text_sha256: "sha256:…"        # hash of the EXACT authoritative text this was made from
generation_method: model              # model | human | hybrid
generator: "claude-haiku-4-5-20251001"  # model+version, or human name
generation_date: "2026-07-05"
model_params: { temperature: 0, prompt_id: "concepts-v1" }
review_status: unreviewed             # unreviewed | machine_only | human_reviewed
reviewed_by: null
license: "CC-BY-4.0"                  # OUR contribution
confidence_note: "Automated first pass; not legally reviewed."
disclaimer: >
  Derived, unofficial content generated by the Space Law Corpus. NOT the
  authoritative legal text. See the linked authoritative source and its
  authentic-language text for anything legally operative.
```

### Why `source_text_sha256` matters
It records the exact fingerprint of the authoritative text the derivation was based on. If the authoritative text is later corrected (a new version), we can instantly detect that every derived artifact built on the old hash is now **stale** and must be regenerated or re-reviewed. This is how the derived layer stays honestly tied to the authoritative layer over time.

## 4. Concept tagging is neutral and enumerated

`concepts.json` tags authoritative text against a fixed, neutral vocabulary — never an advocacy frame. The Phase 1 concept set:

`registration`, `liability`, `jurisdiction_and_control`, `non_appropriation`, `resource_rights`, `abandonment_and_salvage`, `state_responsibility`, `rescue_and_return`, `environmental_and_debris`, `dispute_settlement`.

Each tag links to the specific article/section it applies to (via `structure.json` anchors), with the generating model and review status recorded. Abandonment/salvage is simply one tag among these — present so the corpus can answer questions through that lens, without the corpus being *about* that lens.

## 5. Rules that keep the wall standing

1. **Direction of dependency is one-way.** Derived files reference authoritative files. Authoritative files never reference or depend on derived files.
2. **No model output in `authoritative/`.** Ever. Machine translations, summaries, and tags are derived, full stop.
3. **Regeneration is allowed in `derived/`, forbidden in `authoritative/`.** Derived artifacts can be re-generated with better models and overwritten (their own history stays in Git); authoritative text is append-only and versioned (doc 02).
4. **Every derived artifact is self-labelling.** The `disclaimer` and `derived_type` travel with the file, so even if a single file is copied out of the repo, it still announces that it is derived and unofficial.
5. **Machine-checkable enforcement (Phase F).** A CI validation script will: reject any file under `authoritative/` whose metadata indicates model generation; reject derived artifacts lacking a valid `source_corpus_id`/`source_version_id`/`source_text_sha256`; and flag derived artifacts whose `source_text_sha256` no longer matches the current authoritative text (stale).

## 6. Cost discipline for the derived layer

Mechanical passes (structure extraction, first-pass tagging, draft translations) use cheaper/free models; the output is quarantined here and marked `review_status: unreviewed` until a human review upgrades it. This keeps model work cheap and clearly labelled, and keeps human judgement — the scarce resource — focused where it matters (doc: Phase E/F).

## 7. How a user reads the corpus

- Need the law itself? Read `authoritative/.../text.txt` (and check `original.<ext>` and `metadata.yaml`).
- Want help navigating it? Read `derived/...` — knowing everything there is unofficial and traceable back to a specific authoritative version.

The wall means a reader is never in doubt about which of these they are looking at.

## Phase E refinement (2026-07-02, recorded per doc 02)

When the derived layer was first built, two small, backward-compatible refinements were made to the derived-metadata format (the authoritative schema and the two-layer wall are unchanged):

1. **`derived-metadata.yaml` holds a LIST of artifact records**, one per artifact (e.g., `structure.json`, `concepts.json`), each with an `artifact_file` field — so multiple derived artifacts can coexist in one version folder with independent provenance.
2. **`generation_method` accepts `rule_based`** (in addition to `model`, `human`, `hybrid`) for deterministic, non-model passes. The first structure-extraction and concept-tagging passes are `rule_based` and `review_status: unreviewed` — mechanical first drafts explicitly awaiting human/model review.

Both are reflected in `schema/derived-metadata.schema.json`.
